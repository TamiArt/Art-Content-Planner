# Future Implementation Plan

Этот файл фиксирует задачи, которые осознанно откладываются на будущие итерации, чтобы не терять продуктовые решения и не перегружать текущий этап разработки.

## PWA и полноценная offline-работа

Сейчас приложение работает как локальная SPA без сервера, а service worker оставлен заглушкой. Полноценный offline-режим нужно реализовать позже отдельной задачей.

### Что нужно сделать

- Добавить реальную регистрацию service worker.
- Подключить PWA-инструмент для Vite или написать собственный service worker.
- Настроить кеширование HTML/CSS/JS/assets после сборки.
- Настроить стратегию обновления приложения, чтобы пользователь не оставался на старой закешированной версии.
- Проверить сценарии:
  - первое открытие онлайн;
  - повторное открытие без интернета;
  - обновление версии приложения;
  - сохранность данных в localStorage;
  - импорт/экспорт JSON без сети.
- Добавить понятное UI-уведомление, если доступна новая версия приложения.

### Важно

Offline/PWA не заменяет резервные копии. Экспорт JSON должен остаться обязательной рекомендацией, потому что localStorage может быть очищен браузером или пользователем.

## Картины, услуги и офферы

Пункты меню **Мои картины**, **Услуги** и **Офферы** остаются отдельными разделами. Сейчас нужно поддерживать их как каталоги с ручным добавлением и редактированием, а полноценную связь с генерацией реализовать позже.

### Текущая роль

- **Мои картины** — каталог конкретных работ, которые можно продавать или использовать как визуальные примеры.
- **Услуги** — постоянные направления работы художника.
- **Офферы** — конкретные продающие предложения, акции и упаковки.

### Будущая связь с генератором

Нужно сделать эти блоки источниками для генератора контент-плана:

- Для целей `lead` и `sale` генератор должен уметь выбирать конкретную картину, услугу или оффер.
- Для целей `trust` генератор может использовать услуги как источник экспертного и процессного контента.
- Для целей `reach` и `engagement` картины можно использовать как визуальный повод: цвет, интерьер, до/после, процесс, история создания.

### Связь постов с источниками

Добавить в модель поста связь с коммерческим источником, например:

```ts
sourceType?: 'painting' | 'service' | 'offer';
sourceId?: string;
```

Или более явно:

```ts
relatedPaintingId?: string;
relatedServiceId?: string;
relatedOfferId?: string;
```

### Prompt builder

Расширить prompt builder, чтобы он получал контекст выбранного источника:

```ts
buildPromptForPost(post, settings, {
  painting,
  service,
  offer,
});
```

Для картины передавать:

- название;
- размер;
- технику;
- цену;
- цвета;
- настроение;
- подходящий интерьер;
- ключевую идею;
- CTA.

Для услуги передавать:

- что входит;
- для кого услуга;
- сроки;
- цену;
- требования к клиенту;
- результат;
- CTA.

Для оффера передавать:

- что получает клиент;
- целевую аудиторию;
- боль клиента;
- решение;
- сроки;
- цену;
- CTA-фразу.

### Генератор месяца

Добавить в генератор настройки:

- продвигать картины;
- продвигать услуги;
- продвигать офферы;
- выбирать источники автоматически;
- исключать уже использованные источники на ближайшие N публикаций.

### Редактор поста

Добавить в редактор поста блок “Источник контента”:

- тип источника;
- выбранная картина/услуга/оффер;
- быстрый переход к редактированию источника;
- отображение CTA и ключевых деталей источника.

### Аналитика

После связи постов с источниками добавить аналитику:

- какие картины дают больше заявок;
- какие услуги лучше конвертируют;
- какие офферы дают больше лидов;
- какие форматы лучше продают картины;
- какие CTA работают лучше.

### UX-навигация

Пока оставить три отдельных пункта меню:

- Мои картины;
- Услуги;
- Офферы.

В будущем можно рассмотреть объединение в общий раздел **Продажи** с вкладками, если сайдбар станет перегруженным.
